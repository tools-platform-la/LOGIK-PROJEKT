bash
# Fork on GitHub (web UI), then:
git clone git@github.com:<your-gh-username>/LOGIK-PROJEKT.git

# Point "upstream" at the original

git fetch upstream --prune
git branch -r

Why: youll pull updates from upstream, push your work to your fork (origin).

2) Keep a clean tracking branch for upstreams default

Stick to the repos default branch for syncs (today: release-2026.2.0).

bash
# Create a local branch that tracks upstreams default
git checkout -b release-2026.2.0 upstream/release-2026.2.0

# Publish that branch to your fork (origin) once
git push -u origin release-2026.2.0

3) Do your changes on feature branches

Dont edit the tracking branch directly; branch off it:

bash
git checkout release-2026.2.0
git pull --ff-only upstream release-2026.2.0   # update before branching

# ...edit files, preferably under cfg/site-cfg/ ...

Open a PR from your forks feature branch into your forks release-2026.2.0 (or use it directly if its only for your use).

4) Pull upstream updates over time

When the original updates:

bash
# Update local view of upstream
git fetch upstream --prune

# Fast-forward your local tracking branch
git checkout release-2026.2.0
git merge --ff-only upstream/release-2026.2.0
git push   # update your forks release-2026.2.0 too

# Rebase your feature branch onto the updated base (keeps history clean)
git checkout feat/site-config-overrides
git rebase release-2026.2.0     # resolve any conflicts if prompted
git push --force-with-lease     # update the PR/branch after rebase

If you prefer merges over rebases, replace the last two lines with:

bash
git merge release-2026.2.0
git push

Pattern A moving base that always mirrors upstreams current default

One-time setup:

bash
# clone your fork
git clone git@github.com:<you>/LOGIK-PROJEKT.git
cd LOGIK-PROJEKT

# point to upstream (original repo)
git remote add upstream https://github.com/flamelogik/LOGIK-PROJEKT.git

# record which branch upstream marks as default (upstream/HEAD)
git fetch upstream --prune
git remote set-head upstream -a

# create a local branch 'base' that mirrors upstream's current default
git checkout -B base upstream/HEAD
git push -u origin base

Day-to-day workflow:

bash
# 1) Update 'base' to whatever upstream's default is *today*
git fetch upstream --prune
git remote set-head upstream -a
DEFAULT=$(git symbolic-ref --short refs/remotes/upstream/HEAD | sed 's#^upstream/##')

git checkout base
git reset --hard "upstream/${DEFAULT}"   # base now equals the latest upstream default
git push --force-with-lease              # keep your fork's 'base' in sync

# 2) Start or update your work off 'base'
# new work:
git switch -c feat/site-config-overrides base
# ...edit, commit...
git push -u origin feat/site-config-overrides

# update an existing feature after upstream switches releases:
git checkout feat/site-config-overrides
git rebase base
git push --force-with-lease

Why this works:

  • upstream/HEAD always points at the current default branch (whatever release name they pick next).

  • You keep a single local branch base that you reset to upstream/HEAD. Your feature branches always rebase/merge onto base, so when upstream flips to a new release branch, you rebase onceno renaming or hunting.

Optional hardening:

  • Make base your forks default branch in GitHub settings to keep PRs/visibility consistent.

  • If you prefer merges: replace git rebase base with git merge base and drop the --force-with-lease.

Pattern B carry a small patchset across versioned release branches

If you want per-release snapshots:

bash
git fetch upstream --prune
git checkout -B upstream-release upstream/HEAD       # exact release tip today
git push -u origin upstream-release

# your site changes live in a tiny topic branch you rebase each release
git switch -c site-overrides upstream-release
# ...commit your config-only changes under cfg/site-cfg/...
git push -u origin site-overrides

# when upstream moves to next release:
git fetch upstream --prune
git checkout upstream-release
git reset --hard upstream/HEAD
git push --force-with-lease
git checkout site-overrides
git rebase upstream-release
git push --force-with-lease

Keep conflicts minimal

  • Keep your edits in the intended override locations (e.g., cfg/site-cfg/) and avoid touching core files you dont own.

  • Add machine-local secrets/paths to .gitignore.

  • Enable Gits conflict memory for recurring merges: git config --global rerere.enabled true.

Tiny sync script you can drop into scripts/sync-upstream.sh

bash

git checkout base
git reset --hard "upstream/${DEFAULT}"
git push --force-with-lease
echo "Base synced to upstream/${DEFAULT}"

Use:
